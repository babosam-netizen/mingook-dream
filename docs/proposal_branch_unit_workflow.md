# 제안서: 부서 단위(Branch Unit) 시스템 + 모둠원 기여 워크플로

작성일: 2026-05-05  
상태: **결정 확정 — 구현 대기** (결정 항목 전부 확정됨)

---

## 1. 현재 구조의 문제 진단

### 1.1 워크플로 단절

```
현재:
  모둠원 4명 각자 활동 (댓글·토론)
        ↓  (연결 없음)
  대표 1명이 혼자 BillTemplate / MinisterPolicyDraft / VerdictTemplate 작성
```

- 모둠원이 조사하고 토론한 내용이 최종 문서로 연결되는 경로 없음
- 기여 데이터가 휘발됨 (채팅성 댓글만 남고 구조화된 입력 없음)

### 1.2 역할이 '직함'에 불과
- "내가 이 역할이면 무엇을 작성하는가"와 연결되지 않음

### 1.3 부서 배치 고정 + 외부 피드백 없음
- 교사가 배치 모둠 수 조정 불가
- 배치 안 된 모둠의 역할 미정의
- 외부(시민단·기사단)에서 모둠 작업에 피드백을 줄 수 없음

---

## 2. 설계 목표

1. **메모 → 섹션 → 확정** 3단계 유기적 흐름
2. **역할이 실질적**: 역할마다 담당 섹션 지정
3. **부서 단위 유연화**: 교사가 몇 개 모둠을 각 부서에 배치할지 설정
4. **평가단(시민단·기사단)**: 남은 모둠을 평가단으로 배치 가능 — 외부 피드백이 실시간으로 모둠 작업창 상단에 누적
5. 기존 BillTemplate / MinisterPolicyDraft / VerdictTemplate 최종 출력 폼 재사용

---

## 3. 핵심 개념: 부서 단위(Branch Unit)

> **한 모둠 = 하나의 부서 단위 = 하나의 최종 문서**

```
예시 A — 기본 6모둠
  입법:  모둠1(법안1), 모둠2(법안2)
  행정:  모둠3(환경부), 모둠4(복지부)
  사법:  모둠5(검사팀), 모둠6(변호팀)
  평가단: 없음 (전 모둠 배치됨)

예시 B — 4모둠 배치 + 2모둠 평가단
  입법:  모둠1, 모둠2
  행정:  모둠3
  사법:  모둠4(검사), 모둠5(변호)
  평가단: 모둠6 → 시민단 또는 기사단 배치
```

## 4. 모둠원 3단계 워크플로 (확정)

```
[1단계] 메모 카드         [2단계] 역할별 섹션 초안      [3단계] 대표 최종 확정
  모둠원 4명 각자 작성  →   담당 섹션만 편집 가능     →   대표가 전체 검토·수정·확정
  링크 1개 이상 첨부        비담당 섹션: 읽기 전용        4섹션 모두 "완료" 상태여야
  자유 서술 + URL           대표는 전체 편집 가능          [최종 확정] 버튼 활성화
```

### 4.1 결정 사항 (확정)

| 항목 | 결정 내용 |
|---|---|
| 대표 지정 | 교사가 직접 지정 (역할 배정 화면에서) |
| 섹션 편집 권한 | 담당자만 편집, 비담당자는 읽기 전용. 대표는 전체 편집 가능 |
| 메모 카드 | 필수 단계. 링크(신문·뉴스 URL) 1개 이상 첨부 필수 |
| 최종 확정 조건 | 4개 섹션 모두 "완료" 상태여야 [최종 확정] 활성화 |
| 배치 안 된 모둠 | 평가단(시민단/기사단)으로 배치 가능. 모두 배치되면 평가단 없음 |
| 평가단 기능 표시 | 평가단 존재 시에만 해당 학생에게 UI 표시, 없으면 완전 숨김 |

## 5. 평가단 시스템 (신규 설계)

### 5.1 개요

평가단은 부서에 배치되지 않은 모둠에 부여하는 역할.  
**핵심 원칙**: 평가단의 의견(시민단 요청·기사단 기사)은 해당 모둠의 작업창 상단에 실시간으로 누적되어, 모둠원이 작업하면서 외부 압력을 직접 체감하게 한다.

```
[시민단 요청 ①]  "환경 예산 10% 이상 늘려야 합니다"   ← 시민단 작성
[기사 ②]        "모둠1 법안, 실현가능성 낮다는 평가"   ← 기사단 작성
───────────────────────────────────────────────────────
[모둠 작업 영역]
  메모 카드 / 섹션 초안 / 대표 확정 ...
```

### 5.2 평가단 종류

#### 시민단체(시민단)
- **열람**: 배치된 모든 부서 단위의 작업 내용 전체 읽기 전용
- **댓글 작성**: 특정 부서 단위를 지목해 요청·의견 작성
- **표시 위치**: 해당 부서 단위 작업창 상단 피드에 누적 (최신 → 위)
- **형식**: 단체명 + 요청 메시지 (150자 이내) + 타임스탬프

#### 기사단
- **열람**: 배치된 모든 부서 단위의 작업 내용 전체 읽기 전용
- **기사 작성**: 헤드라인 + 본문(200자 이내) + 대상 부서 단위 지정
- **표시 위치**: 해당 부서 단위 작업창 상단 피드에 누적 (최신 → 위)
- **형식**: 헤드라인(30자) + 본문 + 타임스탬프

> **시민단 요청과 기사단 기사는 같은 상단 피드에 누적**된다.  
> 종류 구분은 뱃지(🏛️ 시민단 / 📰 기사단)로 표시.

### 5.3 상단 피드 UI (모둠 작업창 상단)

```
┌─────────────────────────────────────────────────────────┐
│  📢 외부 의견 (3건)                            [접기 ▲] │
├─────────────────────────────────────────────────────────┤
│  📰 기사단(모둠5) — 14:23                               │
│  "모둠1 환경법안, 예산 근거 빠진 채 발의 예정"           │
│  모둠1이 제출한 법안 초안에 예산 섹션이 비어있어...      │
├─────────────────────────────────────────────────────────┤
│  🏛️ 시민단(모둠6) — 14:18                              │
│  "플라스틱 규제 조항에 중소기업 예외 조항 삭제 요청합니다"│
├─────────────────────────────────────────────────────────┤
│  🏛️ 시민단(모둠6) — 14:05                              │
│  "환경부 예산 최소 15억 이상 책정 요청"                  │
└─────────────────────────────────────────────────────────┘
│  [모둠 작업 영역 — 메모 카드 / 섹션 초안 ...]            │
```

- 피드는 항상 펼쳐진 상태로 시작, 접기 가능
- 새 의견 추가 시 상단에 추가되며 강조 표시(3초)
- 모둠원 작성 중에도 실시간 업데이트

### 5.4 평가단 배치 설정

```
[교사 설정 화면]

┌─ 평가단 배치 ────────────────────────────────────────┐
│  배치 안 된 모둠: 모둠5, 모둠6                         │
│                                                        │
│  모둠5: [시민단 ▼]  대상: [전체 / 입법만 / 행정만 ▼] │
│  모둠6: [기사단 ▼]  대상: [전체 / 입법만 / 행정만 ▼] │
│                                                        │
│  * 배치된 모둠이 없으면 이 섹션 자동 숨김              │
└────────────────────────────────────────────────────────┘
```

- 평가단 모둠은 시민단/기사단 중 하나로 지정
- 대상 범위: 전체 부서 또는 특정 부서만 지목 가능
- 평가단 기능(글쓰기 UI)은 해당 학생에게만 표시

## 6. 역할 재설계 — 섹션 담당 매핑

### 6.1 입법부

| 역할 | 담당 섹션 | 작성 내용 |
|---|---|---|
| **조사역** | `배경·근거` | 문제 현황, 관련 통계, 필요성 |
| **논리역** | `핵심 조항` | 구체적 법 조항 2~3개 |
| **분석역** | `예상 효과` | 기대 효과, 예산 소요 추정 |
| **반박역** | `우려 대응` | 예상 반론과 재반박 |
| **의원대표** | 전체 편집 + 확정 | 4섹션 검토·수정 + 최종 확정 |

### 6.2 행정부

| 역할 | 담당 섹션 | 작성 내용 |
|---|---|---|
| **조사역** | `현황 분석` | 현재 상태, 문제점, 정책 필요성 |
| **시행역** | `시행 방법` | 시행 절차, 담당 기관, 일정 |
| **예산역** | `예산 편성` | 항목별 예산 배분 |
| **위험역** | `부작용 관리` | 예상 부작용, 민원 대응 |
| **장관** | 전체 편집 + 확정 | 4섹션 검토·수정 + 최종 확정 |

### 6.3 사법부 — 검사팀

| 역할 | 담당 섹션 | 작성 내용 |
|---|---|---|
| **사실역** | `사건 개요` | 혐의 사실 정리 |
| **증거역** | `증거 목록` | 물적·인적 증거 |
| **법조역** | `적용 법조항` | 위반 법률 조항 |
| **심문역** | `증인 심문 계획` | 증인 목록, 심문 질문 |
| **검사** | 전체 편집 + 확정 | 4섹션 검토·수정 + 최종 변론 |

### 6.4 사법부 — 변호팀

| 역할 | 담당 섹션 | 작성 내용 |
|---|---|---|
| **반박역** | `검사 주장 반박` | 검사 측 허점, 반박 논리 |
| **정황역** | `정황 설명` | 피고인 입장, 정황 증거 |
| **법조역** | `방어 법조항` | 무죄·감경 근거 법조항 |
| **심문역** | `반대 심문 계획` | 검사 측 증인 반대 심문 |
| **변호사** | 전체 편집 + 확정 | 4섹션 검토·수정 + 최종 변론 |

## 7. RTDB 스키마

### 7.1 config — 부서 배치 설정

```
rooms/{roomCode}/config/
  branchConfig: {
    legislative: {
      units: [
        {
          unitId: 'leg_1',
          groupId: 'group_abc',
          title: '환경보호특별법안',
          representativeStudentId: 'student_X'   ← 교사가 지정
        },
        { unitId: 'leg_2', groupId: 'group_def', title: '노동권보호법안', representativeStudentId: 'student_Y' }
      ]
    },
    executive: {
      units: [
        { unitId: 'exe_1', groupId: 'group_ghi', ministryName: '환경부', representativeStudentId: 'student_Z' },
        { unitId: 'exe_2', groupId: 'group_jkl', ministryName: '복지부', representativeStudentId: 'student_W' }
      ]
    },
    judicial: {
      prosecution: [
        { unitId: 'pro_1', groupId: 'group_mno', representativeStudentId: 'student_V' }
      ],
      defense: [
        { unitId: 'def_1', groupId: 'group_pqr', representativeStudentId: 'student_U' }
      ]
    },
    evaluators: [
      { groupId: 'group_stu', type: 'citizens',  targetScope: 'all' },   // 시민단
      { groupId: 'group_vwx', type: 'press',     targetScope: 'legislative' }  // 기사단
    ]
  }
```

### 7.2 branchDrafts — 모둠 작업 데이터

```
rooms/{roomCode}/branchDrafts/
  {unitId}/                        ← leg_1 / exe_1 / pro_1 / def_1 등
    memberNotes/
      {studentId}: {
        text: '...',
        links: [                   ← 링크 1개 이상 필수
          { url: '...', title: '...', addedAt: timestamp }
        ],
        updatedAt: timestamp
      }
    sections/
      {sectionKey}: {              ← 'background' / 'clause' / 'effect' / 'rebuttal' 등
        content: '',
        authorRole: 'researcher',
        authorStudentId: '',
        status: 'empty|draft|ready',
        updatedAt: timestamp
      }
    finalDoc/
      content: { ... }             ← BillTemplate / MinisterPolicyDraft / VerdictTemplate 구조
      lockedAt: null
      lockedBy: null               ← 대표 studentId
      status: 'unlocked|locked'
```

### 7.3 externalFeed — 평가단 피드 (핵심 신규 경로)

```
rooms/{roomCode}/externalFeed/
  {feedId}: {
    type: 'citizens' | 'press',    ← 시민단 요청 또는 기사단 기사
    authorGroupId: 'group_stu',
    authorStudentId: 'student_A',
    targetUnitId: 'leg_1',         ← 대상 부서 단위 (또는 'all')
    
    // 시민단 요청 필드
    requestText: '...',            ← 150자 이내

    // 기사단 기사 필드
    headline: '...',               ← 30자 이내
    body: '...',                   ← 200자 이내

    createdAt: timestamp,
    isVisible: true                ← 교사가 숨김 처리 가능
  }
```

- 모둠 작업창 구독: `externalFeed` 중 `targetUnitId === 내 unitId || 'all'` 필터
- `createdAt` 역순 정렬 → 최신이 상단
- 실시간 구독 → 새 항목 추가 시 상단에 삽입 + 강조 애니메이션

### 7.4 최종 확정 → 기존 경로 publish

```
branchDrafts/{unitId}/finalDoc  확정 시 →
  입법: bills/{billId}
  행정: policies/{groupId}
  사법: verdicts/{caseId}
```

기존 `bills/` `policies/` `verdicts/` 경로는 최종 산출물 경로로 그대로 유지.

## 12. 구현 단계

### 1단계 — 부서 배치 설정 (교사 → config)
- `branchConfig` RTDB 구조 확정 + `ClassroomConfigEditor` UI
- 입법/행정/사법 탭에서 unitId 수만큼 카드 분기 렌더링

### 2단계 — 메모 카드 + 섹션 초안
- `branchDrafts/{unitId}/memberNotes` 구독·작성 (링크 첨부 포함)
- `scaffolding-data.js` 역할 재정의 + 섹션 키 매핑
- 섹션별 담당자 권한 분기 UI

### 3단계 — 대표 확정 + 기존 폼 연동
- [최종 확정] → `bills/` `policies/` `verdicts/` publish
- 4섹션 완료 조건 체크

### 4단계 — 평가단 피드 시스템
- `externalFeed` RTDB 구독·작성
- `ExternalFeedBar.jsx` — 작업창 상단 실시간 피드
- `EvaluatorPanel.jsx` — 시민단·기사단 작성 UI
- 평가단 배치 없을 시 전체 UI 숨김

## 13. 요약

```
핵심 변화:
  모둠 = 1개 부서 단위 = 1개 최종 문서
  역할 = 담당 섹션 1개 책임 (비담당은 읽기 전용)
  대표 = 교사 지정, 전체 편집 + 최종 확정 권한
  메모 카드 = 필수, 링크 1개 이상

교사 유연성:
  입법 1~4모둠, 행정 1~4모둠, 사법 검사 1~2팀 + 변호 1~2팀

평가단:
  남은 모둠 → 시민단(요청 작성) 또는 기사단(기사 작성)
  두 종류 모두 해당 부서 단위 작업창 상단 피드에 실시간 누적
  모두 배치 시 평가단 없음 → UI 전체 숨김

기존 코드 보존:
  BillTemplate / MinisterPolicyDraft / VerdictTemplate → 최종 출력 폼 유지
  bills/ policies/ verdicts/ 경로 → 최종 산출물 경로 유지
  branchDrafts/ + externalFeed/ → 신규 경로
```
