## 1. 한 줄 요약

**디자인 자체는 매우 완성도가 높고 적용 가능합니다. 다만 ‘리스킨(피부 갈아입기)’이 아니라 ‘구조 확장 + 디자인 시스템 전면 교체’ 수준의 작업입니다.** 시각 톤만 가져올지, 새로 추가된 기능·구조까지 가져올지를 먼저 결정해야 합니다.

## 2. 디자인의 정체성 (DNA 분석)

### 2.1 미감 — 한국 편집 디자인 / 신문·갱지 / 브루탈리스트
- **컬러 팔레트**: 갱지(paper #f3ecd8)·먹(ink #1b1a16)·관인(red #c0362c) + 포스트잇 6종(노랑/초록/파랑/보라/분홍/주황 파스텔)
- **타이포**: 5종 한글·영문 웹폰트 혼용
  - `Black Han Sans` (헤드라인·강조)
  - `Gowun Batang` (제목·인용 — 고딕 명조)
  - `Gaegu` (손글씨 — 학생 작성물 느낌)
  - `IBM Plex Mono` (시간·번호·메타데이터 — 신문 활자 느낌)
  - `Noto Sans KR` (본문)
- **그림자**: `box-shadow: 3px~6px 0 var(--ink)` — 단단한 색면 그림자(브루탈리스트)
- **테두리**: 1.5px 먹선이 모든 면에 일관 적용
- **장식**: 도장(원형 stamp), 회전(`-.3deg` 살짝 비틀린 포스트잇), 줄무늬(공책 줄, 빗금 사진 자리)

### 2.2 핵심 UI 패턴 (재사용 컴포넌트)
1. **Statusbar** (상단 검정 띠): 브랜드 도장 + 페이즈 칩(①②③④) + 타이머 + MY ROLE + 학생 번호 아바타
2. **Ticker** (여론판 무한 스크롤 — NEW v4): 빨간 라벨 + `@keyframes tick` 60s 무한 스크롤 + 우측 [+ 기사 쓰기]
3. **Actsteps** (활동 스텝바): 현재 페이즈 안의 9개 세부 단계 표시 (done/now/upcoming)
4. **Taskstrip** (NOW 띠): 빨간 라벨 + 굵은 한 줄 안내 + 우측 CTA
5. **Sidenav** (좌측 60px 아이콘 메뉴): 광장/선거/국정/여론판/갤러리/도움
6. **Botnav** (하단 페이지 네비): ← 이전 / 활동 X / 9 / 다음 →
7. **포스트잇 카드** (`.pts-card`, `.slogcard`, `.cmt`, `.voices .v`): 6색 변형 + 살짝 회전
8. **도장(stamp)**: 원형 빨간 점선/2중선 테두리, 회전 -8°
9. **Device frame**: 1280×830 고정 박스에 6px 그림자

## 3. 현재 코드와의 호환성 분석

### 3.1 호환 가능 / 그대로 가져올 수 있는 것 ✅
| 항목 | 현재 | 디자인 | 비고 |
|---|---|---|---|
| React 컴포넌트 구조 | 분리됨 | 와이어프레임도 명확히 컴포넌트화 가능 | 1:1 매핑 가능 |
| 컬러 변수 시스템 | tokens.js (PHASE_META, TOPIC_BG) | CSS custom properties | tokens.js에 paper/ink 팔레트 추가만 하면 OK |
| 페이즈 4단계 구조 | `currentPhase 1~4` | 동일 | 그대로 |
| RTDB 데이터 스키마 | rooms/{rc}/... | 와이어프레임은 데이터 명세 없음 | 영향 없음 |
| Firebase 보안 규칙 | v0.8.53 적용 | 영향 없음 | 그대로 |

| 한글 웹폰트 | 시스템 폰트 | 5종 Google Fonts (~400KB+) | 소 — 인덱스 추가, 다만 초기 로딩 무거워짐 |
| 상단 Statusbar | RoomBar (간단한 네비) | 브랜드 + 페이즈칩 + 타이머 + 역할 + 아바타 5분할 | 중 — RoomBar 전면 재설계 |

### 3.3 새로 만들어야 하는 것 / 시스템 구조 변경 🚧
| 항목 | 설명 | 영향 |
|---|---|---|
| **Phase 1 안 9단계 세부 활동** | 현재는 자유 진행 / 디자인은 오프닝→단체결성+슬로건→자료수집→정리→포스터→평가→출력토론→투표→회고 9단계 스텝 | **데이터 구조 추가**: `currentActivity` 또는 `currentSubStep` 필드, 교사 [다음 활동] 신호 |
| **단체 합류 + 3점 분배 메커니즘** | 슬로건 입력 후 모둠 슬로건들에 3점을 나눠 분배 → 1위 슬로건 채택 | **신규 RTDB 경로**: `rooms/{rc}/groups/{gid}/slogans/{sid}/points/{voterId}` |
| **포스터 평가 듀얼모드** | 전체모드(6개 한눈에) ↔ 입력모드(1개 집중 + 다른 의견 + 내 의견) | UI만 / 데이터는 기존 comments 재사용 가능 |
| **Activity Steps Bar** | 현재 페이즈 안 세부 단계 표시 | 페이즈 1만 9단계, 페이즈 2/3/4도 같은 구조로 만들지 결정 필요 |
| **MY ROLE 표시** | 학생이 현재 갖고 있는 역할 (단체·시민·후보·국회의원·기자 등) | scaffolding-data.js의 roles와 연동 필요 |
| **출력·오프라인 토론 활동** | 7번 활동: 포스터 인쇄해 교실에서 토론 | 디지털 단계가 아님 — UI 가이드만 표시? |
| **9. 회고 기사** | 시민광장 끝에서 기사로 마무리 | NewsBoard와 연결 |

## 4. 예상 문제점 및 리스크

### 4.1 큰 문제 ❌
1. **스코프 폭증**: 단순 ‘리스킨’ 아님. v4.1만 해도 Phase 1을 9단계로 재구조화하는 변경. 현재 MVP는 자유 진행 구조. 4페이즈 모두 동일하게 세분화하면 **사실상 v2 빌드** 수준의 재작업.
2. **세부 활동 단계 데이터 모델**: `currentSubStep`을 추가하면 교사 대시보드에 ‘다음 활동’ 버튼 9×4=36개 신호 관리가 필요. 현재 PhaseWorkflow와 충돌 가능.
3. **3점 분배 메커니즘은 새 게임 메커닉**: 데이터 모델·UI·검증 모두 신규. 현재 슬로건은 단순 텍스트. 기존 데이터와 호환성 고민 필요.

### 4.2 중간 문제 ⚠️
4. **웹폰트 5종 로딩 비용**: 한글 폰트 압축돼도 초기 로딩에 ~500KB. 학교 와이파이 환경에서 첫 진입 지연 가능. 폰트 디스플레이 전략(`font-display:swap`) 필수.
5. **Tailwind와 커스텀 CSS 공존**: 디자인은 Tailwind 안 씀(순수 CSS+변수). 전역 CSS 레이어를 새로 두면 현재 Tailwind 컴포넌트와 스타일 충돌 위험. 둘 중 하나로 통일하거나, Tailwind에 디자인 토큰을 이식해 유틸리티로 표현하는 방법 권장.
6. **Statusbar 1280px 가정**: 디자인은 데스크탑 1280×830 고정 캔버스. 학생 노트북·태블릿(iPad 1024×768) 대응이 필요. 반응형 전략 미정의 상태.
7. **데이터 스키마 변경 시 구버전 방 호환**: 이미 v0.8.52에서 한 번 겪은 문제. 9단계 활동 데이터 추가 시 또 폴백 로직 필요.

### 4.3 작은 문제 / 주의 사항 ℹ️
8. 디자인의 `transform:rotate(-.3deg)` 회전은 접근성·모바일 터치 영역에 미세한 영향. 무방하지만 일관성 유지 주의.
9. 포스트잇 색상 6종이 현재 PHASE_META 색과 별개. 충돌 안 나도록 토큰 분리 명확히.
10. 여론판 티커 무한 스크롤 애니메이션은 기존 OpinionSideBanner와 중복. 둘 다 둘지, 하나로 통합할지 결정 필요.

## 5. 좋은 점 (Pro)

✅ **교실 분위기·프로젝트 컨셉과 강하게 정합**: ‘다큐멘터리 제작자 페르소나’, ‘작은 대한민국’이라는 슬로건과 신문·갱지·도장 미감이 완벽히 어울림. 현재의 평범한 Tailwind 룩보다 훨씬 더 ‘민국이의 꿈’답다.
✅ **학생 동기 부여 효과**: 포스트잇·손글씨체·도장 같은 아날로그 장치가 “이건 진짜 활동이다” 느낌을 강화. 디지털 무드보드의 차가움을 상쇄.
✅ **정보 위계가 명확**: Statusbar(상시 컨텍스트) → Ticker(주변 정보) → Actsteps(현재 위치) → Taskstrip(지금 할 일) → 본문 → Botnav(이동) — 학생이 어디 있는지·뭘 해야 할지 늘 알 수 있음. 현재 RoomBar보다 정보 밀도·가시성 모두 우수.
✅ **컴포넌트화 잘 됨**: 와이어프레임이 이미 컴포넌트 단위로 분리되어 있어 React 매핑이 깔끔함.
✅ **자체 보유한 ‘디자인 언어’**: 일관된 컬러·타이포·간격 시스템이 갖춰져 있어, 새 화면 추가 시 디자이너 개입 없이 컴포넌트 조합만으로 확장 가능.

## 6. 단점 (Con)

❌ **‘리스킨’이 아니라 ‘리빌드’ 수준**: 디자인이 아닌 구조까지 변경하는 부분이 많음. 시간 산정 오해 위험.
❌ **고정 1280px 데스크탑 가정**: 태블릿 시나리오(아이패드 + 키보드 케이스 등)에서 깨질 가능성.
❌ **웹폰트 비용 + 초기 로딩**: 학교망에서 폰트 로딩 지연 시 깜빡임(FOUT) 가능.
❌ **9단계 세부 활동 게이팅**: 학생 자율성을 줄이고 교사 신호 의존을 늘림. 현재 교사가 [다음 활동] 신호를 9×4=36회 누르게 되는 운영 부담. 수업 흐름이 더 매끄러울 수도 있고, 더 빡빡해질 수도 있음 — 양면.
❌ **포스트잇 회전 + 손글씨 폰트**: 글이 길거나 한자·영문이 섞이면 가독성이 떨어질 수 있음. 학생 텍스트 길이 제한이 더 엄격해져야 함.

## 7. 권장 적용 전략 (3가지 옵션)

### 옵션 A — 풀 리뉴얼 (권장 X, 이번엔 비추)
모든 와이어프레임을 그대로 구현. **예상 작업: 3~5주 + 데이터 마이그레이션**. 현재 MVP가 동학년 시연을 앞둔 상태에서 이 폭의 변경은 위험.

### 옵션 B — 단계적 적용 (권장 ✅)
**1단계 (1주)**: 디자인 시스템만 이식 — `tokens.js`에 paper/ink/post-it 팔레트 + 5종 웹폰트 + 도장·포스트잇·하드섀도우 유틸 추가. 현재 컴포넌트 골격은 그대로 두고 ‘옷’만 갈아입힘.
**2단계 (1주)**: Statusbar / Ticker / Sidenav를 새로 만들어 RoomBar·OpinionSideBanner와 점진적으로 교체. 페이지별 레이아웃 그대로.
**3단계 (선택)**: Phase 1 안 ‘세부 활동 9단계’와 ‘3점 분배’는 별도 proposal로 분리해 사용자 확정 후 진행. 현재 MVP의 자유 진행 구조와 큰 차이가 있어 시연 후 결정 권장.

### 옵션 C — 미감만 차용 (가장 안전)
컬러·폰트·포스트잇 카드·도장 같은 **장식 요소만** 적용. 9단계 세부 활동, 3점 분배, dual-mode 평가 같은 **구조·메커니즘 변경은 제외**. 작업량 작고 시연 일정에 영향 없음.

---

## 8. 결정이 필요한 항목

1. **적용 범위**: A(풀)/B(단계)/C(미감만) 중 어느 것?
2. **세부 활동 9단계**: Phase 1만 적용? 4페이즈 전체? 아니면 안 함?
3. **3점 슬로건 분배**: 새 메커니즘 도입? 기존 단순 슬로건 입력 유지?
4. **웹폰트 5종**: 전부 로드? 핵심 2~3종(Black Han Sans + Gaegu + IBM Plex Mono)만?
5. **반응형 대응 범위**: 데스크탑만? 태블릿(1024)도? 모바일은 제외?
6. **Tailwind vs 커스텀 CSS**: 전역 CSS 레이어 추가? 토큰을 Tailwind config로 이식?
7. **시연 일정**: 동학년 시연이 언제인지에 따라 옵션 선택이 달라짐.

## 9. 다음 단계 권고

1. 위 결정 항목 7개 중 1~4번을 먼저 확정.
