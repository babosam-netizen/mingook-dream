## 1. 배경과 목표

| **B. 전원 사법 편성** | `all_judicial` | 6모둠 모두 사법 역할 | 모든 학생이 재판에 직접 참여 |

## 2. 사건 시나리오 (두 모드 공통): **"별빛 24시 편의점 시급 인상 사칭 사건"**

> **죄목**: 근로기준법 제36조(금품 청산) 및 제43조(임금 지급 방법) 위반
> 임금체불은 실제로는 단독부 사건이지만, 교육 시뮬레이션 목적으로 "사회적 관심이 큰 사건"으로 가정.

### 2-1. 기본 설정

| 항목 | 내용 |
|---|---|
| 사건명 | 별빛 24시 편의점 시급 인상 사칭 사건 |
| 시기 | 2024년 3월 ~ 12월 |
| 피고인 | **김○○** (45세, 편의점 점주) |
| 피해자 | 알바생 3명 (이○○ 20세, 박○○ 22세, 정○○ 19세) |
| 죄목 | 근로기준법 위반 (누적 미지급 약 700만원) |
| 검사 구형 (예시) | 징역 1년 또는 벌금 2,000만원 |

### 2-2. 사건 경위

> 피고인 김○○은 2024년 1월, 카톡으로 "올해부터 시급을 9,860원(최저시급)으로 올려주겠다"고 약속하였다. 그러나 3월~12월까지 9개월간 시급 8,000원 수준만 지급하였고, 알바생들의 추가 지급 요청을 "장사가 안 돼서"라며 거부하였다. 누적 미지급 임금 약 700만원. 노동청 신고 후에야 200만원만 부분 지급.

**핵심 쟁점**:
1. 인상 약속이 있었는가? (점주: "격려성 멘트")
2. 의도적 미지급인가? (점주: "경영난")
3. 추가 지급 요청을 거부했는가?

### 2-3. 증거 7가지

| # | 증거 | 제출 측 | 핵심 |
|---|---|---|---|
| 1 | 알바생 카톡 인상 약속 캡처 | 검사 | 1월 점주: "시급 9,860원으로 올려줄게~" |
| 2 | 통장 거래 내역 | 검사 | 미지급 누적 약 700만원 |
| 3 | 추가 지급 요청 메시지 | 검사 | 3~10월 알바생들의 정산 요청 3회 |
| 4 | 노동청 조사 결과서 | 검사 | "명백한 차액 미지급" |
| 5 | 매출·영업이익 자료 | **양측** | 검사: "흑자" / 변호: "30% 감소" |
| 6 | 현금 지급 영수증 (점주 자필) | 변호 | 일부 현금 지급 주장 (신빙성↓) |
| 7 | 점주의 회계 자료 (월별 손익) | 변호 | "실제 적자" 입증 시도 |

### 2-4. 피고인 진술서 (시나리오 카드)

> "1월에 알바생들이랑 카톡으로 얘기하다가 '올해는 좀 더 챙겨줄게'라고 한 건 맞아요. 근데 그게 정식 약속은 아니었어요. 그냥 격려 차원이었죠. 3월부터 옆에 대형마트가 들어서면서 매출이 정말 안 좋아졌어요. … 의도적으로 떼먹으려던 건 절대 아닙니다."

## 3. 편성 모드 A: **기자단 편성** (4모둠 사법 + 2모둠 기자단) ⭐ 입법·행정과 동일

### 3-A-1. 모둠 구성 (24명 기준)

| # | 모둠 | 인원 | 역할 분담 |
|---|---|---|---|
| 1 | 🧑‍⚖️ **판사부** | 4명 | 재판장 + 좌배석 + 우배석 + **법정관리관**(기록·진행) |
| 2 | 🧑‍💼 **검사부** | 4명 | 주임검사 + 보조검사 + **알바생A 역할** + **알바생B 역할** |
| 3 | 🧑‍🦱 **변호인부** | 4명 | 주변호사 + 보조변호사 + **점주(피고인) 역할** + **단골손님 역할** |
| 4 | 👥 **배심원부** | 4명 | 배심원장 + 배심원 ×3 |
| 5 | 📰 **기자단 A** | 4명 | 사법기자장 + 사건취재기자 + 인터뷰기자 + 데스크 |
| 6 | 📰 **기자단 B** | 4명 | 동일 구성 (또는 매체 차별 — A는 공식 보도 / B는 SNS·시민 매체) |

→ **24명 표준** / 검사·변호인 모둠 안에 증인·피고인 역할 통합 (v3 패턴 재사용)

### 3-A-2. 기자단 2모둠의 활동

#### 기자단 A (공식 보도)
- ① 단계 → ② 단계: 사건 개요 기사 (재판 전 보도)
- ③ ~ ⑥ 단계: 재판 실시간 보도 (모두진술·증거조사·심문·변론 요지)
- ⑦ 단계: 평결·판결 결과 기사 + 사회적 의미 분석

#### 기자단 B (시민·SNS 매체)
- 같은 단계지만 다른 시각 — 검사 측 vs 변호 측 중 한 쪽 시각, 또는 시민 입장 분석
- 평결 후 시민 인터뷰 기사

→ **두 기자단의 보도 비교**로 학생들이 미디어 관점 차이를 학습

### 3-A-3. 사법 4모둠 안의 역할 흡수 패턴

#### 검사부 4명 안의 [당사자/증인 역할]
- 주임검사·보조검사: 법적 공격 담당
- 알바생A 역할 학생: 9개월 근무, 미지급 350만원, 추가 지급 요청 묵살 경험 증언
- 알바생B 역할 학생: 6개월 근무, "현금 지급은 거짓" 증언

#### 변호인부 4명 안의 [당사자/증인 역할]
- 주변호사·보조변호사: 법적 변호 담당
- 점주(피고인) 역할 학생: 피고인 진술서(§2-4) 기반 답변
- 단골손님 역할 학생: 정상참작 호소 ("평소 좋은 사람이었어요")

> 노동청 조사관·회계사 등 추가 증인은 자료(증거)에 진술서 형태로 포함 (별도 학생 역할 없음)

## 4. 편성 모드 B: **전원 사법 편성** (6모둠 모두 사법 역할)

### 4-B-1. 모둠 구성 (24명 기준)

| # | 모둠 | 인원 | 역할 분담 |
|---|---|---|---|
| 1 | 🧑‍⚖️ **판사부** | 4명 | 재판장 + 좌배석 + 우배석 + 법정관리관 |
| 2 | 🧑‍💼 **검사부** | 4명 | 주임검사 + 보조검사 + 검사보 + 자료조사관 |
| 3 | 🧑‍🦱 **변호사부** | 4명 | 주변호사 + 보조변호사 + 변호사보 + 자료조사관 |
| 4 | 👤 **증인부** | 4명 | 알바생 A + 알바생 B + 알바생 C + 노동청 조사관 (검사측 증인 통합) |
| 5 | 🧑‍💼 **피고인부** | 4명 | 점주 + 배우자 + 회계사 + 친구 (변호측 인적 자원) |
| 6 | 👥 **배심원부** | 4명 | 배심원장 + 배심원 ×3 |

### 4-B-2. 진영 협업 구조

```
┌─ 공소 진영 ────────────┐   ┌─ 변호 진영 ────────────┐
│ 🧑‍💼 검사부 (법적 공격)  │   │ 🧑‍🦱 변호사부 (법적 변호) │
│        ↕ 보조           │   │        ↕ 보조           │
│ 👤 증인부 (사실 증언)    │   │ 🧑‍💼 피고인부 (인적 호소) │
└─────────────────────────┘   └─────────────────────────┘
              ↑                            ↑
              └─── 🧑‍⚖️ 판사부 (절차) ──────┘
                          ↓
                  👥 배심원부 (평결)
```

### 4-B-3. 모드 B의 특징
- 모든 학생이 재판에 직접 참여 (관찰자 없음)
- 증인부·피고인부가 독립 모둠 → 더 사실적인 재판 구조
- 기자단 부재 → 사후 보도/평가 시각 없음 (그 대신 Phase 4 시사회에서 미디어 활동으로 연결)

## 6. 7단계 진행 계획 (두 모드 공통)

| 2 | **② [논고 카드] 작성 (토론 전 준비)** ⭐ | 사법 4모둠은 논고·진술 카드 / 기자단 2모둠은 사전 보도 기사 작성 | 6모둠 모두 논고·진술·평결 준비 카드 작성 |
| 3 | **③ 모두진술** | 검사·변호 모두진술 / 기자단 실시간 보도 시작 | 동일 (단, 기자단 없음) |
| 4 | **④ 증거조사** | 증거 제시 + 모둠 내 [알바생 A·B] 진술 / 기자단 보도 | 검사부↔증인부 협업으로 진술 |
| 5 | **⑤ 피고인·증인 심문** | 변호인 모둠 내 [점주 역할] 학생 답변 / 기자단 인터뷰 | 피고인부의 점주·배우자·회계사 답변 |
| 6 | **⑥ 최종 변론 및 검사 구형** | 검사 구형·변호인 최종변론 / 기자단 사설 작성 | 동일 |

## 14. ⭐ 시나리오 양식 시스템 (v6 신규 — 핵심)

> **목표**: "선생님이 이 양식에만 넣으면, 어떤 사건이든 재판이 자동으로 열린다"
> — 별빛 편의점 사건은 이 양식을 채운 **예시(preset)** 일 뿐이다.

### 14-1. 설계 철학 (v7 — AI 생성 JSON 업로드)

```
┌──────────────────────────────────────────────────────────────┐
│                    사건 시나리오 적용 흐름                      │
│                                                              │
│  ① AI에게 요청       ② JSON 파일 저장    ③ 시스템 업로드     │
│  "이 사건으로        → case.json         → [파일 선택]        │
│   JSON 만들어줘"       (AI가 생성)          [적용하기]         │
│       ↓                    ↓                    ↓            │
│  [프리셋 선택]도 가능 ──────────────────→  재판 자동 진행    │
│   별빛 편의점 / 폐수 방류 / 야간 근무 / 담합 / 개인정보 유출   │
└──────────────────────────────────────────────────────────────┘
```

**핵심 변경 (v6 → v7)**:
- ❌ v6: 교사가 UI 폼에 직접 입력 (항목이 많아 번거로움)
- ✅ v7: AI(Claude·ChatGPT 등)에게 사건 내용을 주고 JSON 파일을 생성받아 업로드

**왜 JSON + AI인가?**
- 재판 시나리오는 항목이 많고 서술형 내용이 길다 → UI 폼보다 AI 생성이 훨씬 빠름
- 생성형 AI가 `JudicialCaseTemplate` 스키마를 보고 곧바로 유효한 JSON을 만들 수 있다
- 선생님이 AI에게 "이 양식으로 만들어줘, 사건은 이거야"라고 하면 30초 안에 완성
- 업로드하면 시스템이 파싱·검증 후 Firebase에 저장 → 재판 즉시 진행 가능
- **코드는 바뀌지 않는다** — 어느 사건이든 JSON 구조가 같으면 작동

### 14-4. 다른 반이 새 사건을 여는 방법 — AI 생성 + JSON 업로드

#### 전체 흐름 (3단계)

```
STEP 1. AI에게 JSON 생성 요청
─────────────────────────────────────────────────
  선생님이 Claude·ChatGPT 등에 붙여넣는 프롬프트:

  "다음 JSON 스키마 형식에 맞게 재판 시나리오를 만들어줘.
   사건 내용: [선생님이 간단히 설명]
   스키마: [아래 14-2의 JSON 스키마를 복사해 붙여넣기]
   결과물은 유효한 JSON 파일로 저장할 수 있게 해줘."

  → AI가 수십 초 안에 완성된 JSON을 출력
  → 파일로 저장 (예: 우리반_환경오염_사건.json)

STEP 2. 시스템에 업로드
─────────────────────────────────────────────────
  학급설정 → 사법부 → 사건 시나리오 탭

  📂 [ JSON 파일 선택... ]  ← input type="file" accept=".json"
  ✅ 유효성 검사 통과 (자동)
  [이 시나리오로 재판 열기]

STEP 3. 자동 적용
─────────────────────────────────────────────────
  → Firebase branchConfig/judicial/activeCase에 저장
  → 7단계 재판이 이 JSON 데이터 기반으로 즉시 작동
```

#### 학급설정 UI 목업

#### AI에게 주는 프롬프트 템플릿 (앱 내 복사 버튼으로 제공)

> 앱의 [📋 스키마 복사하기] 버튼은 이 파일의 프롬프트 전체를 클립보드에 복사한다.

**프롬프트 구조 (요약)**:
```

② [사건 내용] — 선생님이 채우는 부분 (5줄 이내로도 충분)
③ 작성 요구 사항 (고정 — 쉬운 언어·증거 수·증인 수·진술서 형태 등)
④ 출력 형식 — "순수 JSON만 출력"
⑤ JSON 스키마 전체 (빈 껍데기)
```

**선생님이 실제로 바꾸는 부분은 ② 뿐**:
```

- 피고인: 소규모 공장 사장 (50대)
- 피해자: 인근 주민들
- 사건: 폐수를 몰래 하천에 방류하다 적발. "설비 고장으로 몰랐다" 주장
- 관련 법: 환경보전법 (모르면 "적절한 법 적용해줘"라고 써도 됨)
- 분위기: 피고인이 회피하는 편
```
→ 이것만 입력하면 AI가 증거 6개·증인 5명·진술서 전문·단계별 안내 7개를 모두 완성

#### 업로드 처리 로직 (클라이언트 사이드)

```js
// BranchConfigEditor.jsx 내
async function handleCaseJsonUpload(file) {
  const text = await file.text()
  let parsed
  try {
    parsed = JSON.parse(text)
  } catch {
    showError('JSON 파일 형식이 올바르지 않습니다.')
    return
  }

  // 필수 필드 유효성 검사
  const required = ['id','title','caseType','defendant','evidence','witnesses','stageGuides']
  const missing = required.filter(k => !parsed[k])
  if (missing.length) {
    showError(`필수 항목 누락: ${missing.join(', ')}`)
    return
  }

  // stageGuides 7단계 검사
  if (!parsed.stageGuides || parsed.stageGuides.length < 7) {
    showError('stageGuides는 1~7단계 7개가 모두 있어야 합니다.')
    return
  }

  // Firebase에 저장 (파일 업로드 아님 — 파싱 결과를 직접 DB에 씀)
  await set(ref(db, `rooms/${roomCode}/branchConfig/judicial/activeCase`), parsed)
  await set(ref(db, `rooms/${roomCode}/branchConfig/judicial/activeCaseId`), `custom:${parsed.id}`)
  showSuccess(`"${parsed.title}" 사건이 적용되었습니다!`)
}
```

> **파일 저장 없음**: JSON 파일은 서버에 저장하지 않는다. 브라우저에서 파싱 → 검증 → Firebase에 직접 쓰기. NAS `upload.php`는 불필요.

### 14-5. 시스템이 양식을 읽어 재판을 자동 구성하는 흐름

```
교사 입력(양식) → Firebase 저장 → 학생 화면 자동 렌더링

rooms/{roomCode}/branchConfig/judicial/activeCase  ← 현재 적용된 양식 전체
rooms/{roomCode}/branchConfig/judicial/activeCaseId ← 'preset:byeolbit_2024' or 'custom:{roomCode}'

각 단계별 자동 동작:

① 배경 이해 단계
   └─ activeCase.summary + activeCase.charges 표시
   └─ 모둠별 activeCase.roleHints[roleName] 자동 표시

② [논고 카드] 작성
   └─ sectionKey별 드래프트 양식에 roleHints 사전 입력

③ 모두진술
   └─ activeCase.stageGuides[2].studentNote 학생 화면 표시

④ 증거조사
   └─ activeCase.evidence.filter(e => e.revealedAtStage <= 4) 자동 공개
   └─ side별 색상 구분 (검사=파랑, 변호=초록, 양측=보라)

⑤ 피고인·증인 심문
   └─ activeCase.evidence.filter(e => e.revealedAtStage === 5) 추가 공개
   └─ activeCase.witnesses + activeCase.defendant.script 학생 화면 공개

⑥ 최종 변론
   └─ activeCase.prosecutionDemand 자동 표시

⑦ 배심원 평의·판결
   └─ activeCase.keyIssues 체크리스트 형태로 배심원 화면에 표시
   └─ 판결문 작성 양식 자동 생성 (keyIssues 항목별 판단란 포함)
```

## 부록 A. 6인 그룹 확장안

### 모드 A (기자단 편성) — 학급 인원에 따라
| 학급 | 사법 4모둠 | 기자단 | 총 |
|---|---|---|---|
| 20명 | 4×4=16 | 1×4=4 (또는 빼기) | 16-20 |
| 24명 | 4×4=16 | 2×4=8 | **24 (표준)** |
| 28명 | 4×5=20 (각 모둠 +1명) | 2×4=8 | 28 |
| 30명 | 4×5=20 | 2×5=10 | 30 |

### 모드 B (전원 사법) — 학급 인원에 따라
| 학급 | 6모둠 | 총 |
|---|---|---|
| 20명 | 4명×4 + 2명×2(증인·피고인 축소) | 20 |
| 24명 | 4명×6 | **24 (표준)** |
| 28명 | 4명×4 + 6명×2(증인·피고인 확장: 추가 알바생/측근) | 28 |
| 30명 | 5명×6 (각 모둠 +1) | 30 |

### 6인 그룹 추가 역할 후보
- **판사부 +2**: 합의부 보조 판사 + 사법보좌관
- **검사부 +2**: 참여검사 + 사실관계 정리관
- **변호사부 +2**: 추가 변호인 + 정상참작 자료 정리
- **증인부 +2**: 노무사(전문가) + 인근 편의점 점주(변호측 증인)
- **피고인부 +2**: 점주의 부모 + 거래처 사장
- **배심원부 +2**: 배심원 2명 추가
- **기자단 +1**: 사진기자
