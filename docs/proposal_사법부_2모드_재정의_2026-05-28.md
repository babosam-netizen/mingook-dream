## 0. 배경 (설계 맥락)

### 0-1. 교사(사용자)의 최초 구상

사법부 활동을 두 갈래로 구상했다.

- **판결중심모드(시나리오)**: AI가 입법·행정의 결정에 따른 시나리오를 만들되, **역할극 대사**를 작성해 그 대사를 통해 서로의 논점·쟁점이 드러나게 한다. 학생은 재판을 **참관**하고 판결문을 써서 평가·의견을 나눈다.
- **역할중심모드(증거논고)**: AI가 **사건개요·증거·증인**을 세팅해 제시하고, 학생은 맡은 역할에 따라 실제 증거·증인을 근거로 **논고를 작성**한다. 직접 주장·반론하며 재판을 진행한다(배심원 가능).
- 두 모드 모두 **판결은 예시로 존재하되 실제 판결은 학생(판사)의 몫**으로 남긴다.
- 단계 사이마다 교사의 **빠른제어**로 AI 시나리오를 만들고·끼워넣고·진행하며, 도움 내용과 확인사항을 노출한다.

**최초 제시 순서**
- 역할중심: 준비 → 자료조사 → 논고초안 → 온라인토의 → 재판 → 기사 → 판결 여론조사
- 판결중심: 준비 → 쟁점파악 → 온라인토의 → 재판보기 → 판결문작성·게시 → 온라인토의 → 기사정리

### 0-2. 논의로 보강·수정된 점

- **토글 레벨 정렬**: 두 모드는 입법·행정의 `공동작업/역할중심` 토글과 같은 레벨 → 사법부 `workMode` 슬롯을 **판결중심/역할중심**으로 재정의.
- **기존 단계 재사용**: 역할중심의 기사·여론조사는 기존 `article3`·`poll4`를 **사법용 라벨로 껍데기만 씌워** 정식 편입(신규 컴포넌트 X).
- **전원 판사 구조**: 판결중심은 별도 배심원팀 없이, 연기 3팀(판사·검사·변호사) 외 **전 모둠이 판결문**을 쓴다. 연기한 3팀도 판결문을 쓴다.
- **대본 가시성 규칙**: 재판보기에서 대본은 **연기 3팀만 각자 자기 역할 대사**를 본다.
- **대본 데이터 신설**: 판결중심을 위해 사건 JSON에 `trialScript`(speaker별 대사) 필드와 전용 AI 프롬프트·업로드 칸을 추가.
- **재판보기 전 온라인토의 제거(A안, 2026-05-28)**: 재판을 보기 *전* 쟁점 토의는 실제 주장·증거가 없어 근거가 얇다. 온라인토의는 **판결문 게시 후 비교 토의 1회**로 집중하고, 쟁점파악은 "재판을 능동적으로 보게 하는 준비"로 남긴다. 빠진 토의 대신 **판결 여론조사를 마지막 단계로 추가**.

## 1. 핵심 결정 (사용자 확인 완료 2026-05-28)

사법부 활동을 **두 개의 모드**로 정의한다. 이 토글은 입법·행정의 `공동작업 / 역할중심` 토글과 **같은 레벨**이다 (= `branchConfig.judicial.workMode` 슬롯을 재정의).

| 입법·행정 | 사법부 (이번 안) |
|---|---|
| 공동작업 (`collaborative`) | **판결중심 (`verdict`)** |
| 역할중심 (`role`) | **역할중심 (`role`)** |

- 사법부에서 `workMode` 값 = **`'verdict'`(판결중심) / `'role'`(역할중심)**.
- 기존 사법부 `'collaborative'`(모둠 1문서 공동작성)는 토글에서 제외 → 판결중심이 그 슬롯을 차지.
- `mode`(기자단 편성 / 전원 사법) 토글은 그대로 별개 유지.

**두 모드 공통 전제**
- AI 시나리오는 우리 반 통과 법안 + 행정부 시행령 위반 사건에서 생성 (기존 자동 주입 기능 유지).
- 판결(예시)은 존재하되, **실제 판결은 학생의 몫**으로 남긴다. AI는 정답 판결을 주지 않는다.
- 차이는 **"AI가 무엇을 만들어 제시하나"** + **"학생이 무엇을 하나"**.

| 구분 | 역할중심 (증거논고) | 판결중심 (시나리오) |
|---|---|---|
| 학생 위치 | 재판 당사자 (직접 논고·변론·심문) | 관찰자→판단자 (보고 판결문 작성) |
| AI 산출물 | 사건개요 + **증거 + 증인** | 역할극 **대본/대사**(공방 스크립트) |
| 핵심 결과물 | 학생이 쓴 논고·변론 + 실제 재판 | **전 모둠 판결문** + 평가 |
| 현재 구현 | **거의 구축됨**(현 5단계) | **미구현(신규)** |

## 2. 역할중심모드 (증거논고) — 7단계

학생이 검사·변호인·증인·배심원·판사를 직접 맡아, AI가 세팅한 **사건개요·증거·증인**을 근거로 논고를 쓰고 실제 공방을 벌인다.

**AI 산출물**: 사건 개요 + 증거 목록 + 증인 + 역할 힌트 → 현재 `judicial-case-data.js`의 사건 JSON(charges/evidence/witnesses/defendant/roleHints/stageGuides)이 이미 이 모드용. 프리셋 5종 보유.

| # | 단계 ID | 단계 | 현재 상태 |
|---|---|---|---|
| ① | `judicial-prep` | 준비 (교사준비 / 학생: 사건 접수 + 역할 배정) | ✅ 있음 |
| ② | `judicial-research` | 자료조사 (역할별 자료 검색) | ✅ 있음 |
| ③ | `judicial-draft` | 논고초안 (증거·증인 기반 주장 정리) | ✅ 있음 |
| ④ | `judicial-discussion` | 온라인토의 (논고 검토·조언, 여론 탐색) | ✅ 있음 |
| ⑤ | `judicial-trial` | 재판 (주장·반론, 배심원 평결) | ✅ 있음 (토론도구) |
| ⑥ | `judicial-article` | 기사 (사법부가 하는 일 보도) | **기존 article3 재사용 + 사법 라벨** |
| ⑦ | `judicial-poll` | 판결 여론조사 | **기존 poll4 재사용 + 사법 라벨** |

**⑥·⑦ 처리 방식 (확정)**: 새 컴포넌트를 만들지 않는다. 기존 `article3`(기사 작성) / `poll4`(사후 여론조사)의 **기능·데이터 흐름을 그대로 가져오되**, 사법부 단계용 라벨·헤더·안내 문구("사법부가 하는 일 기사", "판결에 대한 시민 여론조사")로 **껍데기만 씌워** 사법 7단계 안에 배치한다.

## 3. 판결중심모드 (시나리오) — 7단계 (신규) · "전원 판사" 구조

학생은 AI 시나리오로 진행되는 재판을 **참관**하고, **모든 모둠이 각자 판결문을 작성**해 게시·평가한다. 별도 배심원팀 없이 **전원이 판사** 역할로 판결문을 낸다.

**AI 산출물**: 역할극 **대본**(판사·검사·변호사 대사). 공방이 대사로 흐르며 논점·쟁점이 자연히 드러나도록 작성. 증거·증인은 학생이 조사하는 게 아니라 대본 안 공방에서 노출. → **사건 JSON에 신규 대본 필드 필요.**

> **단계 구조 확정(2026-05-28, A안)**: 재판을 보기 *전* 온라인토의(쟁점 토의)는 실제 주장·증거가 없어 근거가 얇아 **제거**. 온라인토의는 판결문 게시 후 "판결문 비교 토의" 1회로 집중한다. 쟁점파악(②)은 토의가 아니라 "재판을 능동적으로 보게 하는 준비 활동"으로 남긴다. 빠진 토의 대신 **판결 여론조사를 마지막(⑦) 단계로 추가**.

| # | 단계 ID(안) | 단계 | 비고 |
|---|---|---|---|
| ① | `verdict-prep` | 준비 (교사준비 / 학생: 사건개요 파악 + 팀 배정) | 연기 3팀 지정 |
| ② | `verdict-issues` | 재판 쟁점 파악하기 (재판을 능동적으로 볼 준비) | **신규** 모둠 메모 |
| ③ | `verdict-trial` | 재판보기 (3팀이 대본 연기 / 전 모둠 판결문 메모) | **신규** 대본 송출 |
| ④ | `verdict-writing` | 판결문 작성 (전 모둠) 및 게시 | **신규** 산출물 |
| ⑤ | `verdict-discussion` | 온라인토의 (게시된 판결문 상호 검토) | **핵심 토의 1회** |
| ⑥ | `verdict-article` | 기사 정리 | 기존 기사 재사용 + 사법 라벨 |
| ⑦ | `verdict-poll` | 판결에 대한 여론조사 | 기존 `poll4` 재사용 + 사법 라벨 |

**팀 구성 (확정)**
- ① 준비에서 교사가 **연기 3팀(판사·검사·변호사)** 을 지정. 나머지 모둠은 참관.
- **별도 배심원팀 없음.** 전 모둠이 판결문을 쓰는 "전원 판사" 구조.

**③ 재판보기 — 대본 가시성 규칙 (확정)**
- 대본을 연기하는 **세 팀(판사·검사·변호사)만 각자 자기 역할 대사를 화면에서 본다**. 검사는 검사 대사, 변호사는 변호사 대사, 판사는 판사 대사.
- 나머지 참관 모둠은 대본 없이 연기를 보며 판결문 메모.

**④ 판결문 작성 주체 (확정)**
- **모든 모둠이 판결문을 쓴다.** 연기한 3팀(판사·검사·변호사)도 연기 후 자기 모둠 판결문을 작성한다. (모둠당 판결문 1건)

## 4. 데이터 스키마 변경

### 4-1. `gameStore.js` — `branchConfig.judicial.workMode`
- 값 의미 변경: `'role'`(역할중심) | `'verdict'`(판결중심). 기본값 `'role'`(현 동작 보존).
- **마이그레이션 (확정)**: 기존 `workMode:'collaborative'` 저장 방 → **`'verdict'`(판결중심)으로 매핑**.

### 4-2. 사건 JSON — 대본 필드 신규 (판결중심 전용)
현재 `defendant.script`(피고인 진술)만 있고 공방 전체 대본은 없음. 신규 필드 제안:

```js
trialScript: [
  { order: 1, scene: '모두진술', speaker: 'prosecution', text: '존경하는 재판장님, 본 검사는...' },
  { order: 2, scene: '모두진술', speaker: 'defense',     text: '피고인은 결코 고의로...' },
  { order: 3, scene: '증거조사', speaker: 'judge',       text: '검사 측, 증거를 제시하십시오.' },
  // speaker: 'judge' | 'prosecution' | 'defense' | 'witness' | 'defendant'
]
```
- `speaker`로 필터링 → 각 연기 팀에게 자기 대사만 노출.
- `scene`으로 막 구분(모두진술 / 증거조사 / 증인심문 / 최종변론 등).
- `validateCaseJson`: 판결중심 모드일 때 `trialScript` 존재·speaker 유효성 검사 추가.

### 4-3. Firebase RTDB 신규 키 (판결중심)
- `judicialIssues/{caseId}/{groupId}` — ② 쟁점 메모
- `verdicts/{caseId}/{groupId}` — ④ 모둠별 판결문 (게시·평가)
- ⑤ 온라인토의는 기존 댓글/토의 구조, ⑥ 기사·⑦ 여론조사는 기존 `article3`/`poll4` 구조를 재사용(사법 라벨).
